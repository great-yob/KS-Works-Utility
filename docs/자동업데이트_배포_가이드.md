# 자동 업데이트 & GitHub 배포 가이드

사내 직원 PC에 한 번 설치하면, 이후 새 버전을 GitHub Releases에 올리는 것만으로
모든 PC가 **자동으로 업데이트**됩니다. (electron-updater + electron-builder + GitHub)

---

## ⚠️ 먼저 알아둘 변경점: portable → 설치형(NSIS)

자동 업데이트는 **설치형 빌드에서만** 동작합니다. 그래서 빌드 타깃을 기존
`portable` 에서 **NSIS 설치 프로그램**으로 바꿨습니다 (`package.json` → `build.win.target`).

- `oneClick: true`, `perMachine: false` → **관리자 권한 없이 사용자 폴더(%LOCALAPPDATA%)에
  설치**됩니다. 사내 일반 계정에서도 설치/업데이트가 가능합니다.
- **차등(blockmap) 업데이트 자동 적용** → NSIS 빌드는 `.blockmap` 을 자동 생성하므로,
  업데이트 시 변경된 블록만 내려받습니다. Ghostscript 같은 큰 바이너리가 안 바뀌면
  그 부분은 다시 받지 않습니다. (업데이트 트래픽 최소화 — 별도 옵션 불필요)
- 여전히 단일 설치 파일(`KS-Works-Utility Setup x.y.z.exe`) 하나로 배포됩니다.

> portable 무설치 버전도 같이 내보내고 싶다면 `target: ["nsis", "portable"]` 로 둘 다
> 만들 수 있지만, **자동 업데이트가 적용되는 건 NSIS 설치본뿐**입니다.

## 1. 최초 1회 설정

### 1-1. GitHub 저장소 연결
이 프로젝트는 아직 git 저장소가 아닙니다. 먼저 초기화 후 GitHub에 올립니다.

```powershell
git init
git add .
git commit -m "initial"
git branch -M main
git remote add origin https://github.com/<OWNER>/KS-Works-Utility.git
git push -u origin main
```

### 1-2. `package.json` 의 owner 채우기
`build.publish` 에 placeholder가 있습니다. 본인/회사 GitHub 계정으로 바꾸세요.

```json
"publish": [
  { "provider": "github", "owner": "__GITHUB_OWNER__", "repo": "KS-Works-Utility" }
]
```
→ `"owner": "내깃허브계정"` 으로 수정.

> 사내 비공개로 두려면 **private 저장소**도 가능합니다. 단, 클라이언트가 업데이트를
> 받으려면 토큰이 필요해 설정이 복잡해집니다. 가장 단순한 건 **코드는 private,
> Releases만 public** 인 구조이거나, 전용 public 저장소를 배포용으로 두는 것입니다.

### 1-3. 게시용 토큰 발급
GitHub → Settings → Developer settings → **Personal Access Token** 생성
(`repo` 권한). 발급한 토큰을 빌드 PC의 환경변수로 둡니다.

```powershell
$env:GH_TOKEN = "ghp_xxxxxxxxxxxx"
```

## 2. 새 버전 배포 (매번 반복)

1. **버전 올리기** — `package.json` 의 `"version"` 을 올립니다 (예: `1.0.0` → `1.0.1`).
   자동 업데이트는 이 숫자(semver)를 비교해 동작하므로 **반드시 올려야** 합니다.
2. **빌드 + 게시**:
   ```powershell
   $env:GH_TOKEN = "ghp_xxxx"   # 한 번 설정했으면 생략
   npm run release
   ```
   → `electron-builder --publish always` 가 설치본과 `latest.yml`, `*.blockmap` 을
   GitHub Releases에 자동 업로드합니다.
   빌드 산출물은 `release_build/` 폴더에 생성됩니다(gitignore됨).

> **자동 게시(draft 단계 없음)**: publish 설정에 `"releaseType": "release"` 가 들어
> 있어, 릴리스가 **곧바로 published 상태**로 올라갑니다. 별도의 "Publish release"
> 클릭이 필요 없습니다. (만약 업로드 후 검수 단계를 두고 싶다면 이 값을 `"draft"` 로
> 바꾸면 됩니다 — 그 경우 매번 GitHub에서 수동 게시해야 클라이언트가 봅니다.)

끝. 켜져 있는 모든 직원 PC가 다음 체크(실행 시 + 6시간마다) 때 새 버전을 받아,
앱 종료 시 자동 설치합니다.

## 3. 클라이언트 동작 방식

`electron-main.ts` 의 `setupAutoUpdater()` 가 처리합니다.

- 앱 실행 직후 + 이후 **6시간마다** GitHub Releases를 확인합니다.
- 새 버전이 있으면 **백그라운드로 자동 다운로드**합니다.
  사이드바에 진행률(`업데이트 다운로드 중… NN%`)이 표시됩니다.
- 다운로드 완료 시:
  - 사이드바에 **"업데이트 준비됨 — 재시작"** 버튼이 뜹니다. 누르면 즉시 설치 후 재시작.
  - 누르지 않아도 `autoInstallOnAppQuit = true` 라서 **앱을 끄면 다음 실행 때 자동 적용**됩니다.
- 개발 모드(`app.isPackaged === false`)에서는 자동 체크가 비활성화됩니다.

UI 연결: `preload.ts`(`onUpdateStatus`/`checkForUpdate`/`installUpdate`) →
[src/components/UpdateNotice.tsx](../src/components/UpdateNotice.tsx).

## 4. 코드 서명 (선택, 권장)

서명하지 않은 설치본은 Windows SmartScreen이 "알 수 없는 게시자" 경고를 띄웁니다
(설치/업데이트는 됩니다). 사내 배포라 보통은 그대로 진행해도 무방하지만, 경고를 없애려면:

- 회사 **코드 서명 인증서**(OV/EV)를 발급받아 electron-builder의 `win.certificateFile` /
  `certificatePassword`(또는 환경변수) 로 설정하면 됩니다.
- 서명 시 SmartScreen 평판도 누적되어 경고가 사라집니다.

## 5. 문제 해결

| 증상 | 확인할 것 |
|---|---|
| 업데이트가 안 잡힘 | 릴리스가 **published** 인지, `package.json` version을 올렸는지 |
| 게시 실패 (401/403) | `GH_TOKEN` 유효성 / `repo` 권한 |
| `latest.yml not found` | Releases에 `latest.yml` 이 업로드됐는지(= NSIS 타깃으로 빌드했는지) |
| 매 업데이트가 너무 큼 | `differentialPackage: true` 확인. 큰 네이티브 바이너리는 가급적 버전 간 동일 유지 |
| private repo에서 업데이트 안 됨 | Releases를 public으로 두거나 토큰 기반 피드 구성 필요 |
